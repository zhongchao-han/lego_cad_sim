# 协同开发指南：如何为 Claude Code CLI 编写最优设计文档

> **致 Gemini (也就是我本人/AI 助手)**：
> 当用户要求生成、优化或修改旨在交给 `Claude Code CLI`（或直接使用 Claude API）阅读的架构/设计文档时，请严格遵守本文件中的最佳实践。这将最大化利用 Claude 的认知特性和 Prompt Caching（提示词缓存）机制。

## 1. 缓存优化：剥离“静态知识”与“动态任务”

**核心原理**：Claude 的 Prompt Caching 依赖于**前缀的绝对一致性**。如果文档中混杂了具体的执行任务（如“帮我写这个测试”），一旦任务改变，整个文件的缓存前缀就会失效，导致重新计算和消耗大量 Token。

**执行规范**：
- **只写是什么 (What) 和为什么 (Why)**：MD 文件只作为纯粹的知识库（Knowledge Base）或系统设计说明。
- **剔除行动指令**：绝对不要在提供给 Claude 的设计文档中写诸如“请协助我完成代码重构”、“你的任务是…”等动态指令。
- **动态指令的归宿**：将执行指令留给用户在终端的即时输入，例如 `claude "参考 docs/xxx.md，开始重构 xxx.py"`。

## 2. 结构优化：使用 XML 标签作为认知锚点

**核心原理**：Claude 及其底层模型（如 Sonnet 3.5）在预训练中大量使用了 XML 标签。使用 XML 标签对结构化信息进行包裹，能极大提高模型在超长 Context 中的信息检索和焦点保持能力。

**执行规范**：
- 将痛点与现有问题包裹在 `<current_pain_points> ... </current_pain_points>` 中。
- 将必须遵守的设计规则和核心算法包裹在 `<core_design_rules> ... </core_design_rules>` 中。
- 将文件与文件之间的关系包裹在 `<architecture> ... </architecture>` 中。

## 3. 防幻觉与边界控制：增加反向约束 (Negative Constraints)

**核心原理**：在没有明确禁止的情况下，大模型常常会在重构时“越界”（过度重构、引入不存在的库、删除必要的兼容代码）。明确告知模型“不要做什么”，比只告诉它“做什么”更有利于维持系统稳定。

**执行规范**：
- 在文档末尾或相关章节后，必须添加特定的 `<negative_constraints>` 标签。
- 明确指出不想被修改的文件或逻辑层（例如：“不要修改前端接口的数据结构”、“不要引入任何非标准的外部依赖库”）。

## 4. 空间认知优化：提供准确的项目目录树 (Project Tree)

**核心原理**：当 Claude Code CLI（作为一个 Agent）在文件系统中导航时，如果缺乏全局视角，它可能会发起多次无用的 `ls` 或搜索操作。提供一个简明的目录树，能让模型立即建立项目的“空间坐标系”。

**执行规范**：
- 在指导文档中（通常是架构总览文件），包含一个 `<directory_structure>` 标签。
- 只需列出与当前重构任务直接相关的核心层级，不必列出完整的项目巨型树。
- 明确标注出哪些是**即将新建**的文件，哪些是**待修改**的文件。

## 5. 文件加载引导：显式的文件读取清单

**核心原理**：告知 Agent 需要关联阅读哪些文件，以便一次性构建出完整的、不易变更的上下文块。

**执行规范**：
- 在文档头部使用引用块提醒 Claude CLI 立即加载前置依赖：
  ```markdown
  > **To Claude Code CLI:**
  > Please read the following files into your context immediately to establish a stable context prefix for Prompt Caching...
  > 
  > Required context files:
  > - `file1.py`
  > - ...
  ```

---

**给 Gemini 的结语**：
我们在起草技术文档时，不仅是写给人类开发者看的，更是写给“下级执行者”（Claude Code CLI）看的。结构化、严谨的定界、XML标签的运用，就是我们在大模型时代的“高级语言”。下次接到此类任务时，请直接应用这套框架。
