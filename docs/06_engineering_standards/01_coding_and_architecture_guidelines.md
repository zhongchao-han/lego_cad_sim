# 01. 系统级代码与架构规范红线 (Coding & Architecture Standard)

项目高度讲究静态整洁度防线。所有的代码入库必须无脑、零妥协地遵守以下工程铁律：

## 一、 前端域 (Vite + React + Three.js)

1. **TypeScript 类型防雷阵**：
   - 项目绑定 `@typescript-eslint/recommended-requiring-type-checking` 等级检查。
   - **红线底线：彻查与杜绝 `any`** (`no-explicit-any` 开启并提升至 ERROR)。禁止将前端沦为动态类型狂欢。需要面对未知载荷时，使用 `unknown` 并要求书写运行时的 Type Guard 鉴权逻辑。
   - 不允许任何未使用的变量或导入存在。
2. **状态分离与防穿模机制**：
   - 涉及 Zustand 的 Store 切片，只允许维护纯逻辑映射，禁绝将带有引用的 Three.js 对象实例注入全局响应式流（会引起序列化崩溃和严重的 Proxy 性能逃逸）。对于渲染流实体应依赖 ref。

## 二、 后端核心与物理网关 (FastAPI + Pybullet)

1. **Python 极客静态层 (100% Type Hints)**：
   - 项目被 `mypy --strict` 以及 `ruff` 锁死。
   - 每个函数、每一个生成器，必须携带严格输入签名与 Return Type 暗示。不符合规约则视作垃圾代码。
2. **幂等 (Idempotency) 与防爆重试**：
   - 面向基于状态的业务交互（如构件组合、端口挂载），所有核心写入操作必须实现幂等。确保前端网络延迟引发的双击与重试，永远不会使后台物理网格发生坐标炸裂或叠加挂载。
3. **零容忍异常吞噬**：
   - **绝对禁止编写 `except Exception: pass` 此等埋除错误链的代码**。所有的自定义异常必须实现状态码或明文透传。面对异常情况要求通过回退策略（Fallback）或向上级抛出以激活降级系统。

## 三、 TDD 测试驱动准入准则

1. 所有物理结算（如质心矩阵相乘）、图论算法（拓扑环打断），必须含有独立沙盒 Unit Test，不随组件一起测试。
2. 测试用例除常规成功路径外，强制要求具有对抗边界条件（如零值、超出阈值、网络抖动带来的废包）的破坏性测试（Sad Path）。
