# 03. 可观测性与遥测系统架构规范 (Observability & Telemetry)

一个黑盒物理系统是无法维护的。所有的核心模块都必须贯彻结构化日志与异常链路追踪（Traceability），这是定位和重演线上（或沙盒中）物理崩溃的最后一道防线。

## 一、 日志分级红线

1. **ERROR 级限制**：只当功能发生完全宕机、物理刚体穿透且被捕捉、或发生不可逆转的网络协议熔断时使用。并**必须**附带 Error Traceback 及当时的堆栈快照上下文。
2. **WARN 级原则**：在遇到有回退降级策略（Fallback）的情形（例如由于算力不足导致 IK 步长削减），或非致命的高延时现象时埋点输出。
3. **INFO 级规范**：记载生命周期的挂载、引擎起停、端口注册完毕状态。
4. **DEBUG 级强制注入**：
   - 所有的业务算法层（如拓扑合金融断 `snap_parts`，或者张量矩阵更新步 `stepSimulation` 拦截器），**必须在其所有的核心分支与函数头尾加入 Debug 埋点**。必须无情记录所有的参数输入类型和计算截面的中间变量值。

## 二、 Trace Context 与边界回溯

1. API 接口的每个 Request，需自动生成唯一的 `X-Trace-ID` 并透刷在全生命周期日志中。
2. 若拓扑物理重组失败，Logger 格式必须呈现引发崩溃的 UUID（如目标刚体与母体的 `part_id` 和具体 `port_id`）并落盘，这是要求复盘推导 Bug 的第一铁证。任何没有数据支持的空口猜测，均视为不专业的除错态度。
